# Руководство пользователя

Как работать с AI-агентом для управления проектами: рабочие процессы, команды, практические примеры.

## Начало работы

### Старт сессии

Агент не сохраняет контекст между сессиями. В начале каждого чата укажите:

```
Проект: Мобильное фитнес-приложение
Текущая фаза: Планирование
Утверждённые артефакты: project-charter.md
Текущая задача: сформировать план проекта
```

Если не укажете — агент запросит эту информацию перед началом работы.

### Предоставление данных

Данные можно предоставлять в любом формате: структурированный бриф, свободные заметки, тезисы, ответы на вопросы. Агент самостоятельно извлечёт и структурирует информацию.

Минимум для начала: цель проекта, сроки и хотя бы одно из: бюджет, скоуп, состав команды.

## Жизненный цикл проекта

Агент поддерживает шесть фаз проекта. В MVP активны четыре:

```
Фаза 1: Инициация
  └─ generate-charter → generate-risk-register (initial)

Фаза 2: Планирование
  └─ generate-project-plan → generate-risk-register (detailed) ‖ generate-comm-plan

Фаза 3: Прототипирование / Разработка
  └─ generate-meeting-protocol (независимый, любая фаза)

Фаза 6: Закрытие
  └─ generate-plan-fact-report → generate-closure-report
```

Символ `‖` означает, что скиллы можно запускать параллельно — порядок не важен.

После каждого утверждённого результата агент сообщает, какие задачи стали доступны. Решение о следующем шаге — всегда за вами.

## Справочник скиллов

### generate-charter

Создаёт устав проекта — базовый документ с целями, скоупом, сроками, бюджетом, командой, ограничениями и ключевыми рисками.

**Когда использовать:** В самом начале проекта, когда есть первичные данные.

**Как вызвать:**

- «Сформируй устав проекта»
- «Подготовь устав на основе брифа в знаниях»
- "Generate project charter"

**Что подготовить:** Данные о проекте в любой форме — бриф, ответы заказчика, заметки со встречи, свободное описание.

**Что получите:** Структурированный устав из 8 разделов: цель, скоуп (включено/исключено), сроки с вехами, бюджет, команда и роли, ограничения и допущения, топ-3 риска, блок согласования.

**Что дальше:** После утверждения агент предложит сформировать первичный реестр рисков.

---

### generate-risk-register

Создаёт или обновляет реестр рисков с оценкой вероятности/влияния, матрицей рисков и стратегиями реагирования.

**Два режима:**

- **Первичный** (фаза 1) — на основе устава. 5-10 рисков по нескольким категориям. Владельцы по умолчанию — PM.
- **Детальный** (фаза 2) — обогащает первичный реестр рисками из плана проекта. Привязывает риски к рабочим пакетам, назначает конкретных владельцев.

**Как вызвать:**

- «Сформируй реестр рисков» (первичный)
- «Обнови реестр рисков» (детальный)
- "Generate risk register" / "Update risk register"

**Что подготовить:**

- Первичный: утверждённый устав в знаниях.
- Детальный: устав + план проекта + реестр рисков v1 в знаниях.

**Что получите:** Таблица реестра (ID, риск, категория, вероятность, влияние, уровень, стратегия, действия, владелец, статус) + визуальная матрица рисков.

**Что дальше:**

- После первичного → агент предложит сформировать план проекта.
- После детального → агент предложит сформировать план коммуникаций.

---

### generate-project-plan

Создаёт план проекта с WBS (декомпозиция работ), вехами, зависимостями и картой ресурсов.

**Как вызвать:**

- «Сформируй план проекта»
- «Составь план, оценок длительности нет — оцени по объёму»
- "Generate project plan"

**Что подготовить:** Утверждённый устав в знаниях. Опционально: оценки длительности рабочих пакетов и данные о загрузке команды.

**Что получите:** План с разделами: резюме проекта, таблица фаз и рабочих пакетов (ID, фаза, пакет, ответственный, длительность, даты, зависимости), вехи с критериями завершения, матрица зависимостей, карта ресурсов, допущения и ограничения.

**Особенности:**

- Если нет оценок длительности — агент оценивает самостоятельно и помечает каждую как допущение.
- Если даты выходят за дедлайн из устава — агент отмечает проблемные пакеты и запрашивает решение.
- Рабочих пакетов не более 20 для читаемости.
- Единицы измерения единообразны: рабочие дни (проект до 3 мес.) или недели (более 3 мес.).

**Что дальше:** После утверждения становятся доступны параллельно: детальный реестр рисков и план коммуникаций.

---

### generate-comm-plan

Создаёт план коммуникаций с матрицей стейкхолдеров, расписанием коммуникаций и правилами эскалации.

**Как вызвать:**

- «Сформируй план коммуникаций»
- «Подготовь comm-plan»
- "Generate communication plan"

**Что подготовить:** Утверждённые устав и план проекта в знаниях. Опционально: дополнительные данные о стейкхолдерах и предпочтения по каналам (Slack, Email, Meets и т.д.).

**Что получите:** План с матрицей стейкхолдеров (влияние/интерес), матрицей коммуникаций (кто, что, когда, как), правилами эскалации.

---

### generate-meeting-protocol

Структурирует свободные заметки со встречи в формальный протокол с решениями, задачами и предложениями по изменению плана.

**Как вызвать:**

- «Оформи протокол встречи»
- «Обработай заметки со встречи»
- "Generate meeting protocol"

**Что подготовить:** Заметки со встречи в любом формате — тезисы, свободный текст, копия чата, расшифровка диктофона. Опционально: текущий план проекта (для привязки изменений к рабочим пакетам).

**Что получите:** Протокол с участниками, датой, повесткой, ключевыми обсуждениями, принятыми решениями, задачами (кто, что, когда) и предложениями по изменению плана/документов.

**Примечание:** Этот скилл независимый — используется на любой фазе проекта, сколько угодно раз.

---

### generate-plan-fact-report

Сравнивает плановые и фактические данные проекта по трём измерениям: сроки, бюджет, результаты.

**Как вызвать:**

- «Сформируй план-факт»
- «Сравни план и факт»
- "Plan vs actual report"

**Что подготовить:** План проекта в знаниях + фактические данные в чате (даты завершения вех, фактические затраты, статусы результатов). Агент запросит конкретные данные, если их не хватает.

**Что получите:** Отчёт отклонений: по срокам (каждая веха), по бюджету (каждая статья, если есть данные), статус результатов, общая оценка.

**Что дальше:** После утверждения агент предложит сформировать отчёт о закрытии.

---

### generate-closure-report

Агрегирует все данные проекта в итоговый отчёт о закрытии: результаты, отклонения, реализованные риски, извлечённые уроки.

**Как вызвать:**

- «Сформируй отчёт о закрытии»
- «Закрой проект»
- "Generate closure report"

**Что подготовить:** Устав, отчёт план/факт и извлечённые уроки (в чате — результаты ретроспективы, обратная связь от команды, наблюдения PM). Опционально: реестр рисков (для анализа реализованных рисков) и список открытых вопросов.

**Что получите:** Отчёт с резюме проекта, оценкой достижения целей, анализом отклонений, реализованными рисками, извлечёнными уроками (по категориям), открытыми вопросами и заметками по передаче.

## Работа с агентом

### Процесс ревизий

После того как агент представит документ:

1. **Утвердите** — агент создаст финальную версию и предложит следующую задачу.
2. **Запросите изменения** — опишите, что исправить. Агент доработает и покажет обновлённую версию.
3. После 3 циклов ревизий — агент спросит, продолжать доработку или зафиксировать текущую версию.

### Переключение языка

Пишите на русском — агент отвечает и генерирует документы на русском. Пишите на английском — всё переключается на английский. Шаблоны двуязычные, переключение в середине проекта работает без проблем.

### Передача данных между скиллами

В MVP вы выступаете посредником:

1. Агент генерирует документ.
2. Вы утверждаете его.
3. Загружаете утверждённый документ в раздел Knowledge проекта.
4. Запускаете следующую задачу — агент читает документ из знаний.

### Отслеживание состояния проекта

Ведите файл `project-state.md` для отслеживания статусов артефактов. После каждой задачи агент предложит готовый текст для вставки в этот файл и в лог проекта (`log.md`).

## Рекомендации

**Готовьте входные данные заранее.** Чем больше контекста вы предоставите сразу, тем меньше уточняющих вопросов задаст агент и тем качественнее будет первый черновик.

**Используйте файлы знаний.** Загружайте утверждённые документы в Knowledge, чтобы агент мог обращаться к ним напрямую. Это быстрее и надёжнее, чем вставка в чат.

**Начинайте с устава.** Устав — фундамент, от которого зависят все остальные скиллы. Даже если проект уже зрелый, генерация устава обеспечит консистентность всех последующих документов.

**Внимательно проверяйте допущения.** Агент явно помечает каждое допущение. Это наиболее вероятные точки ошибки — сверьте каждое с реальным контекстом вашего проекта.
