# prd-review-challenger — Руководство пользователя

Адвокат дьявола для продуктовых документов. Вставь PRD или фича-спецификацию — получи структурированный разбор: слабые допущения, открытые вопросы, риски реализации и что упущено.

---

## Как начать

Откройте диалог с Claude и используйте одну из фраз:

```
Проверь мой PRD
Разбери спецификацию
Сыграй адвоката дьявола
Найди дыры в PRD
Покритикуй фича-спек
Что упущено в спецификации
```

Затем вставьте текст документа прямо в диалог.

Claude распознает скил и начнёт рабочий процесс.

---

## Пошаговый сценарий

**Шаг 1 — Вставьте документ.** Скопируйте текст PRD, фича-спецификации или продуктового решения и отправьте в диалог. Добавьте контекст, если он есть: целевая аудитория, стадия проекта, на что обратить особое внимание (UX-риски, техническая сложность, бизнес-допущения).

**Шаг 2 — Claude оценивает тип документа.** Если документ слишком короткий для полноценного разбора, Claude сообщит об этом и даст обратную связь по тому, что есть. Если это технический документ (не PRD), Claude применит тот же критический подход и отметит разницу.

**Шаг 3 — Чеклист полноты.** Claude проверяет наличие: problem statement, целевой аудитории, user stories, acceptance criteria, edge cases, метрик успеха, out-of-scope, технических зависимостей, rollback-плана, влияния на другие части продукта.

**Шаг 4 — Разбор по четырём осям.** Claude выдаёт структурированный ревью в четырёх секциях (см. «Что вы получите»).

**Шаг 5 — Топ-3 приоритета.** Claude заканчивает тремя самыми критичными вещами, которые нужно закрыть до начала разработки.

## Примеры

### Пример 1: Проверка перед презентацией команде

**Вы пишете:**
> «Проверь мой PRD. Строим онбординг для новых B2B-клиентов, запуск через 6 недель. [текст PRD]»

**Claude отвечает:**
> «Слабые места и неявные допущения: документ предполагает, что все новые пользователи — технические ЛПР, но не описывает сценарий, когда admin приглашает нетехнического коллегу. Этот путь не задокументирован...»
> [продолжает по всем пяти секциям]

### Пример 2: Валидация обновлённой спецификации перед кикоффом

**Вы пишете:**
> «Сыграй адвоката дьявола для этой спеки — обновили после дизайн-ревью. [текст]»

**Claude отвечает:**
> «Acceptance criteria в разделе 3 конфликтуют с edge case в разделе 5: когда у пользователя нулевая история сессий, fallback из AC-3 не соответствует поведению в EC-2...»
> [продолжает с полным разбором]

## Что вы получите

Структурированный ревью-отчёт из пяти секций:

**1. Слабые места и неявные допущения** — конкретные утверждения в документе, принятые как данность, но не обоснованные, с прямыми отсылками к фрагментам текста.

**2. Открытые вопросы** — то, на что PRD не отвечает, но обязательно спросят разработчики и дизайнеры (например, «Что происходит при [граничном условии]?»).

**3. Риски реализации и UX** — технические риски (сложность, зависимости, производительность, безопасность) и UX-риски (неочевидное поведение, friction, граничные случаи в интерфейсе) с последствием, если не закрыть.

**4. Альтернативные решения** — 2–3 других способа решить задачу с явными trade-offs.

**5. Чеклист полноты** — таблица с оценкой каждого компонента (✓/✗/△) и топ-3 приоритета для исправления до начала разработки.

## Советы для лучших результатов

Давайте контекст заранее. Чем больше Claude знает об аудитории продукта, ограничениях и стадии проекта, тем точнее критика. Одна строка «B2B SaaS, SMB-сегмент, 6 недель» меняет качество разбора.

Указывайте угол. Если сейчас важны только технические риски, скажите об этом: «Сфокусируйся на рисках реализации». По умолчанию скил покрывает все четыре оси.

Запускайте на ранних черновиках. Чем раньше провести разбор, тем дешевле исправить. Ждать, пока документ «готов» — лишает смысла.

Не отметайте каждое замечание. Часть наблюдений покажется очевидной задним числом — это и есть ценность: вы находите их до разработки.

## Частые вопросы

**В: Можно вставить PRD на английском, если работаю на русском?**
О: Да. Скил отвечает на языке вставленного документа. Если нужен ответ на другом языке — укажите это явно.

**В: PRD конфиденциальный — безопасно ли вставлять его?**
О: Скил работает внутри вашей сессии Claude. Применяются стандартные политики обработки данных Anthropic. Для внутренних документов с повышенной секретностью уточните политику компании по использованию AI-инструментов.

**В: Скил может сразу исправить PRD по своей критике?**
О: Нет — скил критикует, не переписывает. Это намеренно: редактирование — задача PM. Для написания или переработки PRD используйте скил `product-management:write-spec`.

---

## Ограничения

- Не пишет, не переписывает и не улучшает PRD — только критика
- Не знает специфики конкретного продукта без переданного контекста
- Не заменяет ревью с реальными разработчиками, дизайнерами и стейкхолдерами
- Очень короткие документы (менее 3 абзацев) получат частичную обратную связь с замечанием о структурных пробелах
- Не даёт юридической, финансовой или экспертной технической оценки без соответствующего контекста
