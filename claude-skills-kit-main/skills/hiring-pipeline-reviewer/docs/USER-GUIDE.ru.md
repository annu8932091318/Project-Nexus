# Руководство пользователя: Hiring Pipeline Reviewer

Как использовать Hiring Pipeline Reviewer для превращения заметок о собеседованиях в структурированный еженедельный отчёт по кандидатам.

---

## Быстрый старт

Самый быстрый способ получить отчёт по пайплайну:

1. Скопируйте заметки о собеседованиях за прошедшую неделю (любой формат)
2. Напишите «Статус по кандидатам» и вставьте заметки
3. Получите таблицу статусов с флагами и рекомендациями

**Результат:** Структурированный отчёт с этапом, оценкой и следующим шагом для каждого кандидата.

**Время:** ~3 минуты

---

## Сценарии

### Сценарий 1: Подготовка к еженедельному синку с рекрутером

**Ситуация:**

Вы — нанимающий менеджер, готовитесь к пятничному синку с рекрутером. За прошедшую неделю ваша команда провела собеседования с 4 кандидатами на разных этапах. Заметки разбросаны по разным местам — часть подробная, часть из двух строк. Вам нужен единый статус для обсуждения с рекрутером и согласования следующих шагов.

**Что делать:**

1. Соберите все заметки о собеседованиях за неделю
   - Откройте свои заметки и скопируйте всё за последние 7 дней
   - Не беспокойтесь о форматировании — черновые заметки тоже подойдут

2. Запустите скил, написав: «Статус по кандидатам»
   - Если нанимаете по нескольким позициям, уточните: «Статус по кандидатам на роль старшего дизайнера»
   - Вставьте или загрузите заметки

3. Изучите таблицу кандидатов
   - Проверьте текущий этап и дату последнего действия для каждого
   - Обратите внимание на флаги `⚠ застрял` — с ними нужно разобраться сегодня

4. Используйте раздел «Рекомендации» на синке с рекрутером
   - Поделитесь экраном и пройдитесь по рекомендациям
   - Пункты «Нужно решение» становятся повесткой для обсуждения

**Ожидаемый результат:**

Вы получаете одностраничный отчёт с:
- Таблицей кандидатов (этап, оценка, следующий шаг, флаги для всех 4 человек)
- Разделом «Флаги» (застрявшие и кандидаты с неполными данными)
- Рекомендациями (продвинуть / отклонить / нужно решение для каждого)

На синк вы приходите подготовленными, с чёткой повесткой на 30 минут.

**Почему это работает:** Вместо того чтобы разбираться в разрозненных заметках прямо во время встречи, вы приходите с чётким видением пайплайна. Рекрутер получает направление, а не просто статусный апдейт.

---

### Сценарий 2: Принятие решения после финальных интервью

**Ситуация:**

Вы — тимлид, только что завершили полный цикл собеседований на позицию бэкенд-инженера. Трое интервьюеров прислали оценочные листы в общий документ. Форматы разные: один использовал шкалу 1–5, другой — текстовые комментарии. Нужно свести картину вместе перед принятием решения и отправкой результатов в HR.

**Что делать:**

1. Скопируйте все три оценочных листа из общего документа
   - Если есть имена рецензентов — включите их (помогает правильно атрибутировать замечания)
   - Вставьте всё одним блоком

2. Напишите: «Сводка по кандидатам — финальный раунд, бэкенд-инженер»
   - Скил объединит заметки нескольких рецензентов в одну запись по каждому кандидату

3. Проверьте сводную запись
   - Поле оценки содержит либо числовой балл, либо качественное резюме
   - Сильные стороны и замечания объединены от всех рецензентов
   - Противоречивые сигналы видны сразу

4. Примите решение
   - Используйте раздел «Рекомендации» как основу для решения, которое отправите в HR
   - Если команда разделилась, скил выставит флаг «Нужно решение» — запланируйте 30-минутный дебриф

**Ожидаемый результат:**

Вы получаете чистую сводную запись по каждому кандидату с объединёнными заметками рецензентов. Раздел «Рекомендации» — отправная точка для решения.

**Почему это работает:** Вместо ручного сравнения трёх разных оценочных листов вы получаете единый взгляд за секунды. Расхождения во мнениях выявляются автоматически — вы сразу знаете, что обсуждать на дебрифе.

---

### Сценарий 3: Возврат после отпуска

**Ситуация:**

Вы — директор, вернулись из двухнедельного отпуска. За это время по нескольким позициям шла активная работа, и рекрутер оставил вам трёхстраничный документ с заметками о собеседованиях за последние две недели. Нужно быстро понять, где находится каждый кандидат и кто требует срочного решения, пока не потерял интерес.

**Что делать:**

1. Откройте документ рекрутера и скопируйте все заметки
   - Заметки охватывают несколько позиций и нескольких кандидатов

2. Напишите: «Ревью пайплайна найма — итоги за 2 недели»
   - Если полезно, укажите период: «Заметки с 2 по 16 апреля»
   - Скил сгруппирует кандидатов и выявит всех, по кому нет активности

3. В первую очередь смотрите на раздел «Флаги»
   - `⚠ застрял` — кандидаты с наибольшим риском выхода из процесса
   - Расставьте приоритеты: застрявшие сначала, потом «нужно решение»

4. Отправьте точечные действия
   - По застрявшим: свяжитесь с рекрутером по следующему шагу
   - По «продвинуть»: сразу дайте одобрение на следующий этап
   - По «отклонить»: передайте рекрутеру список для уведомления

**Ожидаемый результат:**

Вы получаете полную картину пайплайна по всем активным кандидатам с чёткими флагами приоритетности. Скил определяет, кто рискует «остыть» и по кому можно принять решение прямо сейчас.

**Почему это работает:** Вместо того чтобы читать три страницы заметок и строить мысленную модель, вы получаете приоритизированный список действий за несколько минут. Срочные ситуации (застрявший кандидат на стадии оффера) всплывают сразу.

## Советы

### Совет 1: Подписывайте кандидатов в заметках

Скил может разобрать неразмеченные заметки, но лучше всего работает, когда каждый раздел начинается с имени кандидата. Даже простое «**Иван Петров:**» перед блоком значительно повышает точность. Если используете только инициалы (например, «И.П.»), скил выставит флаг неоднозначности — лучше решить это сразу.

**Профи-совет:** В общем документе создайте простой шаблон: «Кандидат: [Имя] | Этап: [этап] | Дата: [дата]» в начале каждого раздела. Скил распознаёт это мгновенно.

### Совет 2: Укажите критерии найма для точных рекомендаций

По умолчанию скил выводит рекомендации на основе ваших заметок. Для более чётких рекомендаций добавьте строку вроде: «Нанимаем старшего PM, обязательно: технический бэкграунд, опыт кросс-функционального взаимодействия, работа с данными.» Это помогает скилу явно указать на несоответствия, а не просто перечислить замечания.

**Профи-совет:** Добавьте критерии прямо в запрос: «Статус по кандидатам на старшего PM — обязательно: технический бэкграунд, данные, 5+ лет кросс-функционального опыта.»

### Совет 3: Флаг «застрял» — ваша очередь приоритетов

Флаг `⚠ застрял` — самый оперативный сигнал. Кандидаты, застрявшие на стадии оффера, рискуют больше всего: они могут принять предложение конкурента, пока вы думаете. Расставьте недельные приоритеты по типу флагов: сначала застрявшие офферы, потом застрявшие после финальных интервью, затем более ранние этапы.

**Профи-совет:** Если нужен более короткий порог для стадии оффера, укажите это в запросе: «Помечай всех на стадии оффера, у кого нет активности 3+ дня.»
