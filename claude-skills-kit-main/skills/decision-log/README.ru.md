# decision-log — скил Claude для извлечения и ведения лога решений

> Автоматически извлекает структурированные решения из заметок встреч, Slack-тредов и email-цепочек — и ведёт чистый лог отдельно от action items.

---

## Что делает скил

**decision-log** анализирует любой неструктурированный текст (заметки со встречи, Slack-тред, email-переписка) и извлекает все решения — не action items, не информацию, а именно точки принятия решений. Для каждого решения фиксируются шесть полей:

- Что решили
- Почему решили (контекст и причина)
- Какие альтернативы рассматривались
- Кто участвовал
- Когда решение принято
- Связанные задачи или риски

Скил работает в двух режимах: создание нового лога с нуля и пополнение существующего с автоматической дедупликацией.

---

## Как это работает

1. Вставьте текст заметок со встречи или тред
2. Опционально — передайте существующий decision log (для режима append)
3. Выберите формат вывода: таблица или карточки
4. Получите чистый структурированный лог для вставки в Notion, Confluence или любой .md-файл

---

## Быстрый старт

Откройте диалог с Claude и напишите:

- `«Зафиксируй решения из этой встречи: [текст заметок]»`
- `«Извлеки решения из этого тред Slack»`
- `«Обнови мой decision log решениями из сегодняшнего звонка»`
- `«Decision log из этих meeting notes»`

## Примеры

**Вход:**
> Решили использовать PostgreSQL вместо MongoDB — у команды есть экспертиза. Алексей и Мария согласны. Redis рассматривали как кэш-слой, но договорились, что это не в скоупе сейчас.

**Выход (формат карточки):**
```
### Решение 1: PostgreSQL как основная база данных

- **Decision:** Использовать PostgreSQL вместо MongoDB
- **Context:** Команда имеет экспертизу в PostgreSQL
- **Alternatives:** MongoDB рассматривалась, отвергнута; Redis для кэша — вне скоупа
- **Participants:** Алексей, Мария
- **Date:** —
- **Related:** —
```

## Зачем отдельный скил для решений?

Action items и решения — разные вещи. Action items отслеживают, что кто-то сделает. Решения фиксируют, о чём договорились и почему. Большинство AI-инструментов для встреч извлекают только action items. Через полгода никто не помнит, почему был выбран тот или иной технический подход. decision-log фиксирует эту институциональную память автоматически.
