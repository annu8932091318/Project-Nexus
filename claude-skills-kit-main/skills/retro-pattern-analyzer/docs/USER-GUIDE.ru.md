# Руководство пользователя: Retro Pattern Analyzer

Как использовать Retro Pattern Analyzer, чтобы выяснить, что постоянно возвращается в ретроспективах вашей команды — и наконец с этим что-то сделать.

---

## Быстрый старт

Самый быстрый способ получить первый отчёт:

1. Соберите 3–5 файлов ретроспектив `.md` или `.txt` из последних спринтов в одну папку
2. Скажите: «Проанализируй ретро» и укажите путь к папке
3. Получите `retro-patterns-YYYY-MM-DD.md` с повторяющимися болями, нерешёнными задачами и позитивными паттернами

**Результат:** Одностраничный отчёт: какие проблемы возвращаются снова и снова, какие action items так и не были закрыты, что команда стабильно делает хорошо.

**Время:** ~3 минуты

---

## Сценарии

### Сценарий 1: Подготовка к квартальной ретроспективе

**Ситуация:**

Вы scrum-мастер или менеджер команды. Команда завершила Q1 (4 спринта). Вы готовите квартальную ретроспективу и хотите открыть встречу с данными, а не ощущениями. Нужно знать, какие проблемы сохранялись весь квартал, какие action items были подняты, но так и не закрыты, и что стоит перестать обсуждать и просто починить.

**Что делать:**

1. Соберите файлы ретро за Q1
   - Найдите 4 файла заметок из спринтов (любой формат — экспорт из Notion, markdown, plain text)
   - Назовите их последовательно: `retro-s1.md`, `retro-s2.md`, `retro-s3.md`, `retro-s4.md`
   - Положите в одну папку

2. Запустите скил: «Проанализируй ретро» или «Паттерны в ретроспективах»
   - Укажите путь к папке
   - Опционально добавьте: «фокус на командных процессах»

3. Изучите блок «Повторяющиеся боли»
   - Пункты с пометкой `4/4` встречались в каждом спринте — это главные темы для обсуждения
   - Пункты с трендом `↑ растёт` — проблема усугубляется, требует немедленного внимания
   - Пункты с трендом `↓ решается` — ситуация улучшается, отметьте это на ретро

4. Подготовьте открытие ретро
   - Скопируйте таблицу «Повторяющиеся боли» как первый слайд или раздел документа
   - Начните с: «Вот что повторялось весь квартал — сегодня договоримся, что конкретно починим»

**Ожидаемый результат:**

Вы получите отчёт, например:
- «Медленный деплой» — 4/4 спринта, тренд `↑ растёт`
- «Непонятная ответственность за QA» — 2/4 спринта, тренд `→ стабильно`
- Action item «автоматизировать пайплайн» поднят в S1, повторился как проблема в S2, S3, S4

Вы открываете ретро с конкретными данными вместо апелляций к памяти. Команда тратит время на решение правильных проблем, а не на спор о том, существуют ли они.

**Почему это работает:** Команды склонны повторять одни и те же жалобы, не отслеживая их между спринтами. Этот отчёт делает паттерн неоспоримым — и создаёт импульс для его закрытия.

### Сценарий 2: Проверка выполнения action items прошлого спринта

**Ситуация:**

Вы team lead. Два спринта назад команда взяла 3 конкретных action items для улучшения процесса code review. Теперь вы хотите проверить: они были реализованы? Или те же проблемы появились снова?

**Что делать:**

1. Соберите файлы ретро последних 3 спринтов
   - Включите спринт, в котором были созданы action items, и 2 следующих
   - Подходит любой формат, включая грубые заметки

2. Скажите: «Выяви паттерны в ретро»
   - Предоставьте 3 файла

3. Проверьте блок «Нерешённые action items»
   - Ищите action items из спринта N, которые вернулись как проблемы в N+1 или N+2
   - Каждая строка показывает: что это была за задача, когда поднята, когда вернулась

4. Используйте результат на следующем 1-on-1 или командной встрече
   - По нерешённым пунктам: «Что помешало это закрыть?»
   - По решённым (не появились в «Повторяющихся болях»): отметьте прогресс

**Ожидаемый результат:**

Чёткий список action items со статусом выполнения. Например:
- «Установить SLA на code review» — поднят в С5, вернулся в С6, С7 → не решён
- «Добавить автоматический linting» — поднят в С5, в С6 и С7 не появился → скорее всего решён

У вас есть конкретная тема для следующей командной встречи. Не нужно полагаться на память или перечитывать 3 комплекта заметок.

**Почему это работает:** Большинство команд забывают свои action items к следующему спринту. Блок «Нерешённые action items» делает follow-through видимым и конкретным.

### Сценарий 3: Отчёт о здоровье команды для руководства

**Ситуация:**

Вы VP Engineering или руководитель отдела. Вы хотите дать руководству компании краткий обзор трендов здоровья команды за 6 спринтов — не анекдоты, а паттерны. Заметки из ретро есть, но структурированного способа их обобщить — нет.

**Что делать:**

1. Соберите 5–8 файлов ретро за прошлый квартал
   - Попросите scrum-мастеров поделиться заметками в `.md` или `.txt`
   - Нормализуйте имена файлов с датами: `retro-2026-01-15.md`, `retro-2026-01-29.md` и т.д.

2. Скажите: «Проанализируй ретро» или «Повторяющиеся темы в ретроспективах»
   - Укажите путь к папке или список файлов

3. Изучите все три блока отчёта
   - **Повторяющиеся боли:** какие системные проблемы существуют? Как долго они сохраняются?
   - **Нерешённые action items:** что команда обещала, но не сделала?
   - **Стабильные позитивные паттерны:** что команда стабильно делает хорошо?

4. Представьте отчёт руководству
   - Скопируйте три таблицы в презентацию или общий документ
   - Добавьте нарратив: «Главная системная проблема этого квартала — X, встречалась в 5/6 спринтов. Команда брала задачу в спринте 3, но проблема вернулась. Предлагаемое решение: [ваша рекомендация].»

**Ожидаемый результат:**

Одностраничное резюме паттернов здоровья команды — готово для слайда или асинхронного апдейта. Вы представляете данные, а не мнения, и можете указать конкретные спринты, в которых проблемы начались или разрешились.

**Почему это работает:** Руководство не читает заметки из ретро. Им нужны сигналы. Этот отчёт даёт нужный уровень абстракции — устойчивые паттерны, а не разовые жалобы.

## Советы

### Совет 1: Называйте файлы с датами для автоматической сортировки

Скил использует имена файлов для определения хронологии спринтов. Если называть файлы с датами — `retro-2026-03-15.md`, `retro-2026-03-29.md` — скил автоматически их упорядочит и точно покажет тренды. Если файлы без дат, добавьте дату в первую строку каждого файла (`Дата спринта: 2026-03-15`).

**Про-совет:** Договоритесь о единой конвенции именования в команде. Даже простой формат `sprint-N-retro.md` работает, если N последовательный.

### Совет 2: Не переживайте из-за разного формата ретро

Скорее всего, ваша команда не пишет ретро в одном формате каждый спринт. Одни используют «Что мешало / Что помогало», другие — «Start-Stop-Continue» или «Plus-Delta». Скил нормализует все распространённые форматы. Просто передайте файлы как есть.

**Про-совет:** Если файл совсем без заголовков секций (только сырые заметки), скил всё равно обработает его через ключевые слова — но результат будет менее точным. Добавить минимальные заголовки (`# Проблемы` и `# Что хорошо`) занимает 30 секунд и заметно улучшает качество анализа.

### Совет 3: Используйте фильтр по фокусу для адресного анализа

Если вас интересует только одна категория проблем — технический долг, командная коммуникация или процесс релиза — добавьте фразу-фокус при запуске скила: «Проанализируй ретро, фокус на теме релизов и деплоя». Скил отфильтрует отчёт и покажет только темы, связанные с этой областью. Удобно при подготовке к конкретной встрече (например, с техническим лидом или с продукт-менеджером).

**Про-совет:** Запустите отчёт дважды: один раз без фильтра (полная картина), второй раз с фокусом (для конкретной аудитории). Две минуты — два разных разговора.
